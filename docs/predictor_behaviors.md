# LLM Predictor Behavior Patterns

Analysis of `predict(**features)` functions generated by gpt-5.5 and gemini-3.5-flash across 64 benchmark runs (2 models × 16 datasets × 2 runs each). Datasets are from OpenML: adult, bank-marketing, car, credit-g, heart-statlog, hepatitis, kr-vs-kp, lymph, monks-2, mushroom, nursery, soybean, spotify-genre, tic-tac-toe, vote, zoo. Accuracy figures are fraction correct on the evaluation set (2000 rows for most datasets, full dataset for some).

---

## 1. Nearest-Neighbour / Training Set Memorisation

The LLM embeds training examples as inline constants and performs lookup or distance-based voting at predict time. Only gpt-5.5 did this, and only on the `lymph` dataset — presumably because lymph is tiny (148 rows, 18 features) and the full training CSV fits in the generated function.

### 1.1 lymph | gpt-5.5 | ACC: 0.865 | `lymph-gpt-5.5-2ac8f11bb2c45552`

Embeds ~100 training rows as a CSV string, parses it at call time into a list of `(features_dict, label)` tuples, then runs weighted k-NN (k=11) with per-feature weights. Uses an exact-match fast path before falling back to distance-based voting. Prior bias is injected into the vote counts:

```python
data = """lymphatics,block_of_affere,...,class
arched,yes,...,2
...
"""

votes = {0: 0.03, 1: 0.47, 2: 0.62, 3: 0.01}   # prior bias
for label, dist in neighbors:
    votes[label] += 1.0 / (0.1 + dist)
return max(votes, key=votes.get)
```

What's notable: the prior is calibrated to the training class distribution (class 2 dominates), not uniform. Distance is a custom weighted Hamming-style metric with per-feature weights chosen by the LLM.

### 1.2 lymph | gpt-5.5 | ACC: 0.919 | `lymph-gpt-5.5-a8051d75ed65b229`

Second run. Same overall approach but refined: adds an exact-match dict lookup as the first step (O(1) before the O(n) k-NN scan), uses k=9, and changes the kernel to `1/(0.25+d)`. Accuracy improves by ~5 percentage points over run 1.

```python
exact = {}   # built once from training CSV
for row in training_rows:
    key = tuple(sorted(row.items()))
    exact[key] = row["class"]

query_key = tuple(sorted(input_features.items()))
if query_key in exact:
    return exact[query_key]
# else: k-NN fallback
```

Two runs on the same dataset produce structurally identical strategies but differ in k, kernel bandwidth, and fast-path implementation — suggesting the LLM is stochastically exploring a known pattern rather than re-deriving it each time.

## 2. Pure Mathematical Formulas

The function body is a small closed-form expression with no lookup tables and no branching beyond a single threshold test. Rare — requires the LLM to recognise that the target is a mathematical function of the inputs.

### 2.1 monks-2 | gpt-5.5 | ACC: 1.0 | `monks-2-gpt-5.5-aac05cc6feec7eb0`

The MONKS-2 concept is: "exactly two of the six attributes equal 1". gpt-5.5 found it:

```python
def predict(**features):
    vals = [
        features.get("v1"), features.get("v2"), features.get("v3"),
        features.get("v4"), features.get("v5"), features.get("v6"),
    ]
    count = sum(1 for v in vals if str(v).strip() == "1")
    return 1 if count == 2 else 0
```

Five lines of logic, perfect accuracy (ACC 1.0) on all 432 rows. Both gpt-5.5 runs and both gemini-3.5-flash runs achieved 1.0 on this dataset, making monks-2 the only dataset where both models solved it perfectly across all four runs.

### 2.2 monks-2 | gemini-3.5-flash | ACC: 1.0 | `monks-2-vertex_ai-gemini-3.5-flash-*`

Same concept independently derived; structurally identical `count == 2` check. Confirmed across both runs.

## 3. Lookup Tables / Rule Tables

Hardcoded dicts mapping individual feature values to scores or class labels. Differs from decision trees in that the branching is data-driven (explicit enumeration) rather than threshold-based.

### 3.1 lymph | gemini-3.5-flash | ACC: 0.919 | `lymph-vertex_ai-gemini-3.5-flash-93c85cc010b98b47`

Log-likelihood ratio (LLR) scoring that reads like manually computed Naive Bayes. Each feature value has a hardcoded contribution to a running log-odds score:

```python
lymphatics_map = {"arched": -0.24, "deformed": 0.08, "displaced": 0.35}
block_of_affere_map = {"yes": -0.64, "no": 0.89}
extrav_lymphatics_map = {"yes": 0.15, "no": -0.19}
...
score = -0.28   # log prior

for feature_map, feat_name in zip(all_maps, all_names):
    val = str(features.get(feat_name, "")).lower()
    score += feature_map.get(val, 0.0)

# classify by sign of accumulated score
```

The values are consistent with log-odds estimated from training frequencies. This is effectively hand-coded Naive Bayes without writing the Bayes math explicitly — the LLM collapsed training statistics into magic constants. Achieved the same accuracy (0.919) as gpt-5.5's k-NN on the same dataset.

### 3.2 vote | gemini-3.5-flash | ACC: 0.917 | `vote-vertex_ai-gemini-3.5-flash-4429917b075914a7`

Single weighted score over 16 binary vote features. The comment in the generated code explicitly labels the approach "Naive Bayes log-odds weights":

```python
weights = {
    'physician_fee_freeze': 6.9,
    'adoption_of_the_budget_resolution': -4.4,
    'crime': 4.5,
    'el_salvador_aid': 4.3,
    'mx_missile': -4.0,
    ...
    'water_project_cost_sharing': 0.0
}
score = -0.45   # intercept / prior log-odds
for feature_name, weight in weights.items():
    val_str = str(features.get(feature_name, "")).lower()
    if val_str in ('y', 'yea', 'yes', '1', 'true', 't'):
        score += weight
    elif val_str in ('n', 'nay', 'no', '0', 'false', 'f'):
        score -= weight
return 1 if score >= 0 else 0
```

Physician fee freeze dominates (weight 6.9) — historically the most partisan issue in the 1984 House. The `water_project_cost_sharing` feature gets weight 0.0 (correctly identified as non-informative). This is compact and accurate.

### 3.3 soybean | gpt-5.5 | ACC: 0.819 | `soybean-gpt-5.5-20cfdf6936fbd5d9`

Multi-class soft voting over 19 disease classes. Each symptom feature updates multiple class scores simultaneously:

```python
scores = {i: 0.0 for i in range(19)}

stem_cankers = features.get("stem_cankers", "")
if stem_cankers in ("present", "above_soil"):
    scores[0] += 1.2; scores[5] += 0.8; scores[12] += 0.6
elif stem_cankers == "below_soil":
    scores[3] += 1.4; scores[8] += 0.7

# ... ~200 lines of similar feature→class score updates ...

return max(scores, key=scores.get)
```

The approach is correct in structure but the weights are imprecise, hence 0.819 rather than 1.0. For 19 classes this is a reasonable result.

## 4. Decision Tree Style

Explicit `if/elif` chains on feature values or thresholds, encoding a decision tree. Both models use this heavily; the difference is depth, coverage, and whether the tree is exhaustive or approximate.

### 4.1 car | gpt-5.5 | ACC: 1.0 | `car-gpt-5.5-4a642f5c3a7db492`

The car dataset has a known exact rule set. gpt-5.5 encoded it perfectly using helper normalisation functions:

```python
def _safety_level(v):
    v = str(v).strip().lower()
    if v in ("low",): return 0
    if v in ("med", "medium"): return 1
    if v in ("high",): return 2
    return -1

safety = _safety_level(features.get("safety"))
persons = _persons_level(features.get("persons"))

if safety == 0 or persons == 0:   # safety=low or persons=2
    return 2   # unacceptable
```

Both gpt-5.5 runs achieved 1.0 on car. The helpers (`_price_level`, `_doors_level`, `_persons_level`, `_lug_level`, `_safety_level`) convert freeform string inputs to ordinal integers, then the main logic is a compact decision tree over those integers.

### 4.2 tic-tac-toe | gpt-5.5 and gemini-3.5-flash | ACC: 1.0 (all 4 runs)

All four runs solved tic-tac-toe perfectly with structurally identical logic: enumerate the 8 winning lines as index tuples, check if all three squares equal `"x"`, return 1 if any line wins. The only variation is whether normalisation handles edge-case inputs ("1", "player x", "crosses", etc.). gpt-5.5 run 1 includes an unusually exhaustive normalisation dict:

```python
x_values = {"x", "player x", "player_x", "cross", "crosses", "xx", "1"}
o_values = {"o", "0", "zero", "player o", "player_o", "nought", "noughts",
            "ought", "oughts", "circle", "circles"}
blank_values = {"b", "blank", "empty", "none", "null", "na", "n/a", "-", "_", ".", " ", ""}
```

### 4.3 zoo | gpt-5.5 | ACC: 0.962 (both runs)

Clean hierarchical decision tree over biological boolean features. Correct ordering: feathers → birds; milk → mammals; fins → fish; etc. Near-perfect accuracy on 101 rows. Both runs are structurally similar but diverge in tie-breaking for reptiles/amphibians (the hard cases).

### 4.4 zoo | gemini-3.5-flash | ACC: 0.962 and 0.885

Run 1 (0.962) gets the taxonomy right. Run 2 (0.885) contains a comment block where the LLM visibly confuses the class-number mapping (`"Note on target mapping: 0 (Mammal), 1 (Bird), 2 (Reptile)..."`) but then writes code that uses a different mapping than the comment describes. The confusion causes misclassifications in the reptile/amphibian boundary. Interesting case where the comment reveals LLM uncertainty that the code then propagates incorrectly.

### 4.5 mushroom | gpt-5.5 | ACC: 1.0 (both runs)

Odor-first decision tree. All mushrooms with odor in `{c, f, m, p, s, y}` are poisonous (return 1); odor in `{a, l}` is edible (return 0); odor `n` falls through to a fallback on `spore_print_color`, `gill_size`, `stalk_surface_above_ring`. Both runs solved it perfectly — odor is a near-perfect discriminator in this dataset.

### 4.6 vote | gemini-3.5-flash | ACC: 0.899 | `vote-vertex_ai-gemini-3.5-flash-7630536e533c6116`

Pure decision tree using only 4 of 16 features. Physician fee freeze → budget resolution → El Salvador aid:

```python
if p_fee == 'y':
    if budget == 'y':
        return 0   # moderate Democrat
    return 1       # standard Republican
elif p_fee == 'n':
    if budget == 'n':
        if salvador == 'y' or education == 'y':
            return 1
        return 0
    return 0       # standard Democrat
```

Accurate but lower than the weighted-score approaches (0.899 vs 0.917) because ignoring 12 features loses discriminating power for ambiguous cases.

## 5. Hybrid Strategies

A hard override rule (exploiting a strong discriminating feature) followed by a fallback formula or scoring function. The most common pattern in gpt-5.5 functions for numeric/mixed datasets.

### 5.1 adult | gpt-5.5 | ACC: 0.846 | `adult-gpt-5.5-692ccc2b9c4db2bf`

Hard rule → weighted score. The hard rule captures the clearest signal immediately:

```python
capital_gain = features.get("capital_gain", 0)
if capital_gain >= 7000:
    return 1    # almost certainly >50K

score = -3.7    # start biased toward 0 (<=50K majority class)

education_map = {
    "doctorate": 4.1, "prof-school": 3.8, "masters": 2.9,
    "bachelors": 1.8, "some-college": 0.4, "hs-grad": -0.2,
    "11th": -0.9, ...
}
occupation_score = {
    "exec-managerial": 2.4, "prof-specialty": 2.1,
    "tech-support": 0.8, "craft-repair": -0.2,
    "handlers-cleaners": -1.5, ...
}
# ... feature accumulation ...
# Interaction terms at end:
if marital in married_vals and occupation in high_skill_occ:
    score += 1.2
```

The approach is calibrated against known income predictors. Synonym dicts (`education_map`, `occupation_score`, `workclass_score`) handle categorical string variants.

### 5.2 adult | gpt-5.5 | ACC: 0.854 | `adult-gpt-5.5-d05acd9f4c76dc46`

Second run. Same strategy but module-level constants (`EDUCATION_TO_NUM`, `HIGH_INCOME_COUNTRIES`), score starts at -2.2 (less pessimistic prior), threshold at 0.75 instead of 0.0. Slightly higher accuracy.

### 5.3 mushroom | gemini-3.5-flash | ACC: 0.837 (2000-row) / lower on full 8124 rows | `mushroom-vertex_ai-gemini-3.5-flash-ab299b7207d57fcb`

Same odor-first logic as gpt-5.5 for the easy cases, but the fallback for odor=`"n"` is oversimplified: `if gill_size == "n": return 1`. This rule is too aggressive — it catches some poisonous mushrooms but also misclassifies edible ones, causing the function to fail on the full 8124-row dataset while appearing fine on a 2000-row sample. Classic case of an LLM over-fitting the fallback branch on a small sample.

### 5.4 hepatitis | gemini-3.5-flash | ACC: 0.846 | `hepatitis-vertex_ai-gemini-3.5-flash-0ad0145e52f221d3`

Hard override exploiting a dataset bias, then scoring fallback:

```python
sex = str(features.get("sex", "")).strip().lower()
if sex == "male":
    return 1   # live (hard rule; males dominate the "live" class in this dataset)

# ... scoring for other features ...
```

The hepatitis dataset is predominantly male patients, and the male→survive correlation is strong in the training data. This is technically correct given the training distribution but is a spurious correlation (a biological sex effect conflated with selection bias in the patient cohort). The LLM is not wrong to use it — it's just exploiting dataset bias.

### 5.5 spotify-genre | gpt-5.5 | ACC: 0.490 | `spotify-genre-gpt-5.5-44e92d8c562050e9`

Three-tier hybrid: artist name lookup → Spanish/text keyword scoring → audio feature scoring. The artist lookup section is the most elaborate code in the entire benchmark — hundreds of artist names hardcoded into sets by genre:

```python
artist_groups = {
    0: ["avicii", "tiësto", "hardwell", "martin garrix", ...],   # EDM
    1: ["j balvin", "maluma", "shakira", "ozuna", ...],           # Latin
    2: ["taylor swift", "katy perry", "dua lipa", ...],           # Pop
    3: ["usher", "beyoncé", "frank ocean", "sza", ...],           # R&B
    4: ["drake", "kendrick lamar", "eminem", "cardi b", ...],     # Rap
    5: ["queen", "nirvana", "metallica", "foo fighters", ...]     # Rock
}
```

If artist matches, apply a large score boost. Then text keyword matching (Spanish words list), then Spotify audio features (speechiness, instrumentalness, energy, tempo ranges per genre). Despite the sophistication, accuracy tops out at ~0.49 because track genre is inherently ambiguous and the dataset contains many artist/track combinations not in the hardcoded lists.

## 6. Surprising or Creative Approaches

### 6.1 credit-g | gemini-3.5-flash | ACC: 0.704 | `credit-g-vertex_ai-gemini-3.5-flash-653a1cb8be85930f`

The generated function contains inline verification comments showing the LLM checking its own rule logic against training data examples:

```python
# Let's verify: "<0,24,existing paid,furniture/equipment,3345,... 0>"
# "<0,12,existing paid,new car,1228,... 0>"
# checking_status == '<0' means less than 0 DM in checking account → bad credit risk
if checking_status in ("<0", "no_checking"):
    ...
```

The LLM appears to have literally traced individual training rows in its comments to verify the rule direction. This is reasoning-in-code: the comments are not documentation, they're step-by-step self-verification that leaked into the output.

### 6.2 vote | gpt-5.5 | ACC: 0.927 | `vote-gpt-5.5-b75571570fd0fd5e`

Two-tier strategy: primary scoring formula plus a post-hoc "confidence override" check. After computing a raw score from all 16 vote weights, the function adds a confirmation gate: if `physician_fee_freeze == "y"` AND at least 4 of 7 other Republican-correlated votes also match, hard-return Republican regardless of the formula score:

```python
if votes["physician_fee_freeze"] == "y":
    republican_signals = 0
    for name, expected in (
        ("adoption_of_the_budget_resolution", "n"),
        ("el_salvador_aid", "y"),
        ("anti_satellite_test_ban", "n"),
        ...
    ):
        if votes.get(name) == expected:
            republican_signals += 1
    if republican_signals >= 4:
        return 1
```

This is an unusual structure: formula first, then signal-count override that can override the formula. It's a hand-coded ensemble of two classifiers in one function.

### 6.3 adult | gemini-3.5-flash | ACC: 0.662 | `adult-vertex_ai-gemini-3.5-flash-1dc77cf424e7db8b`

Instead of bucketed scoring, uses a continuous linear formula with a logit variable:

```python
logit = -5.5
logit += 0.04 * val_age
logit += 0.00003 * val_capital_gain
logit += education_logit_map.get(education, 0.0)
...
return 1 if logit > 0 else 0
```

This is effectively a hand-coded logistic regression. The coefficient `0.04 * age` is a linear term, not bucketed. Interesting because it's conceptually closer to how a trained LR model would look, but the coefficient values are guessed rather than fitted. Accuracy (0.662) is lower than gpt-5.5's bucketed hybrid (0.854), suggesting the continuous linear approach isn't better than bucketed scoring here.

## 7. Failures / Degenerate Cases

### 7.1 car | gemini-3.5-flash | ACC: 0.197 | `car-vertex_ai-gemini-3.5-flash-a7406aba4b4ab4ae`

Near-degenerate on a 4-class dataset (random chance ≈ 0.25). The ordinal mappings for `buying`, `maint`, `lug_boot`, `safety` are correct, but the class label assignments are inverted. The function returns class 2 for "good" conditions when the dataset's class 2 encodes "unacceptable". A class-index confusion error made at construction time that cannot be diagnosed without seeing the training set label encoding.

### 7.2 kr-vs-kp | gpt-5.5 | ACC: 0.524 | `kr-vs-kp-gpt-5.5-b73f873b4a587eab`

Near-chance on a 36-feature chess endgame dataset (binary, so 0.52 ≈ random). The function has a weighted scoring structure but the weights are poorly calibrated. Chess endgame features have highly non-linear interactions that a linear scoring function cannot capture without the LLM having genuine chess knowledge.

### 7.3 kr-vs-kp | gemini-3.5-flash | ACC: 0.551 | `kr-vs-kp-vertex_ai-gemini-3.5-flash-8660c0d95382123f`

Even simpler: sparse if/elif rules followed by `return 1` as the default. The default alone gives 0.518 (the majority-class fraction). The few explicit rules add marginal gain. Both models completely fail this dataset.

### 7.4 nursery | gemini-3.5-flash | ACC: 0.561 | `nursery-vertex_ai-gemini-3.5-flash-be25af271f97a6a9`

The nursery dataset has 5 ordinal features with clear importance ordering. The generated function misidentifies feature semantics — using `v7 == 3` as a test for "child health = very critical" when the actual feature encoding differs. Likely caused by the model confusing feature positions in a positional context with named feature semantics.

### 7.5 soybean | gemini-3.5-flash | ACC: 0.421 | `soybean-vertex_ai-gemini-3.5-flash-1687b0cccdae5672`

On an 18-class classification problem, 0.421 is poor (random chance ≈ 0.056, so it's better than random but far below gpt-5.5's 0.819). The function uses simple if/elif chains that miss most disease classes entirely, falling back to `return 1` for unrecognised patterns. Multi-class problems with many rare classes expose the limits of simple rule enumeration.

## Cross-Cutting Observations

**Model strategy divergence.** gpt-5.5 consistently produces longer, more complex functions with richer synonym handling, interaction terms, and multi-step hybrid logic. gemini-3.5-flash functions are shorter and either use pure scoring (vote, lymph LLR) or simple decision trees. The gpt-5.5 functions look like they were written by someone who read the feature documentation carefully; gemini-3.5-flash functions look like quick heuristics.

**Accuracy gap.** On most datasets gpt-5.5 outperforms gemini-3.5-flash, sometimes substantially: adult (0.854 vs 0.662), car (1.0 vs 0.197), nursery (~0.85 vs 0.561), soybean (0.819 vs 0.421). The exceptions are datasets with clean rules that both models find (monks-2, tic-tac-toe, zoo) or where the LLR approach matches k-NN (lymph, both 0.919).

**Synonym normalisation is a distinguishing trait of gpt-5.5.** Every gpt-5.5 function on categorical datasets includes elaborate normalisation: `_norm_vote()`, `education_map`, `normalize_square()`. gemini-3.5-flash normalisation is minimal (basic `str().lower()` checks). This matters because the benchmark evaluates on the actual dataset strings, so it probably does not directly affect accuracy, but it suggests gpt-5.5 is generating more robust production-ready code.

**The k-NN strategy is unique to gpt-5.5 on small datasets.** Only gpt-5.5 ever embedded training rows — and only on `lymph` (148 rows). This strategy is not feasible for larger datasets (adult at 48000+ rows) and was not attempted there.

**Perfect accuracy is achievable when the concept is a formula or a strict rule set.** monks-2 (count == 2), tic-tac-toe (win-check), mushroom with gpt-5.5 (odor-first), car with gpt-5.5 (exact rule encoding) all hit 1.0. These are datasets where the ground truth is a deterministic function of the features and the LLM can identify it.

**High-dimensional or relational datasets defeat both models.** kr-vs-kp (36 binary chess features, non-linear interactions) causes both models to produce near-random predictors. The LLMs have no chess knowledge and cannot infer the non-obvious feature interactions from samples.

**gemini-3.5-flash makes class-label confusion errors; gpt-5.5 does not.** The car dataset failure (ACC 0.197) and zoo run 2 (ACC 0.885, lower than run 1's 0.962) are both caused by incorrect class-index assignments. The comment in zoo run 2 (`"Note on target mapping: 0 (Mammal), 1 (Bird)..."`) shows the model was uncertain about the label encoding and made the wrong choice. gpt-5.5 was not observed making this error.

**Dataset size has minimal effect on strategy.** The primary driver of strategy choice is whether the LLM can recognise the underlying concept (formula, game rules, biological taxonomy) rather than dataset size. lymph is tiny (148 rows) and got the most elaborate strategy (k-NN); adult is large (48000+ rows) and got the second most elaborate (hybrid scoring).

**The spotify-genre dataset is the hardest.** Best accuracy across all 4 runs is gpt-5.5's 0.508 (run 2) and 0.490 (run 1); gemini-3.5-flash reaches only 0.408–0.411. The text-based artist lookup strategy (gpt-5.5) is the most elaborate approach in the entire benchmark and still barely beats random (6-class, random ≈ 0.167, but class distributions are unequal). Genre classification requires knowledge that neither model's generated code can encode precisely.
